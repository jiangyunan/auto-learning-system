# 网页完整内容导出设计

## 概述

当前 URL 处理链路会在 `ProcessResult.original_content` 中保留抓取后的网页正文，但导出器只输出学习笔记，不输出原始网页正文。因此最终文档内容明显偏少，即使爬虫已经抓到了较完整的页面内容。

本次设计的目标是让默认导出结果变为“学习笔记 + 完整网页正文”，并对正文做轻量 Markdown 优化，在尽量不改变现有抓取和摘要流程的前提下提升最终文档的完整度与可读性。

## 需求

1. 默认导出结果必须包含完整网页正文，而不只是摘要。
2. 导出的正文需要保留基本结构，至少保证标题、列表、代码块、链接等内容可读。
3. 导出的正文需要做保守的降噪处理，尽量去除导航、版权、返回顶部、Cookie 提示等明显噪音。
4. 学习笔记仍然保留在正文之前，维持当前阅读顺序。
5. 不引入新的正文提取依赖，不重写抓取链路，优先做最小正确改动。

## 根因分析

### 现状

1. `src/crawler/__init__.py` 已经通过 `_extract_content()` 提取网页主要文本。
2. `src/pipeline.py` 在构建 `ProcessResult` 时已经将 `doc.content` 写入 `original_content`。
3. `src/exporter/__init__.py` 当前只导出 `l2_summary` 相关内容，不导出 `original_content`。

### 结论

用户看到“生成的文档内容太少”，主因不是导出前的摘要压缩过度，而是导出阶段直接丢弃了已抓取的网页正文。

## 方案选择

### 方案 1：仅在导出阶段追加完整正文并做轻量格式化

优点：

- 改动最小，直接修复当前根因。
- 不影响 chunk 和 summarize 的输入。
- 风险集中在 exporter，测试成本最低。

缺点：

- 对复杂网页的正文完整度提升有限，依赖现有 `_extract_content()` 质量。

### 方案 2：在爬取阶段直接转换为 Markdown 再进入后续流程

优点：

- 数据流更统一，后续导出天然更贴近 Markdown。

缺点：

- 会改变摘要输入形态，可能影响现有摘要质量。
- 改动面扩大到 crawler、chunker、summarizer，超出本次最小修复范围。

### 方案 3：引入更强的正文抽取或 HTML 转 Markdown 依赖

优点：

- 对复杂网页可能有更好的正文保留效果。

缺点：

- 会新增依赖和验证成本。
- 当前问题尚未证明必须通过重构抓取链路解决。

### 选择

采用方案 1。先修复导出阶段遗漏原文的问题，并通过轻量格式化提升 Markdown 可读性。若后续仍有站点出现正文缺失，再单独评估升级正文提取能力。

## 详细设计

### 1. 导出结构调整

`MarkdownExporter` 与 `ObsidianExporter` 统一改为以下顺序：

1. 标题与元数据
2. 学习笔记
3. 原文正文

原文正文新增固定章节：

```markdown
## 原文正文

[格式化后的完整网页正文]
```

行为约束：

1. 当 `result.original_content` 为空时，不输出该章节。
2. 不移除现有学习笔记区块。
3. 不把原文正文插入到 frontmatter 中。

### 2. 正文 Markdown 优化

在 `src/exporter/__init__.py` 内新增轻量私有格式化函数，优先维持最小边界，不单独新增复杂模块。

建议函数边界：

```python
def _format_original_content(self, content: str) -> str:
    ...
```

该函数负责三类处理：

#### 2.1 空白与段落规整

1. 去掉首尾空白。
2. 合并连续 3 个以上空行为 2 个空行。
3. 压缩连续重复行，避免页面采集中的重复段落直接堆叠。

#### 2.2 结构保留

1. 保留已存在的 Markdown 标题、列表、代码块和链接。
2. 对普通正文，保持段落间空行，避免导出成一整块长文本。
3. 对明显的短标题行，可按保守规则提升为 Markdown 标题，但仅在规则足够明确时处理。
4. 不主动生成复杂表格转换逻辑，本次只保证表格文本不被破坏。

#### 2.3 轻量降噪

按行过滤明显噪音，采用保守规则，避免误删正文。优先处理以下内容：

1. 纯导航类短句，如“Home”“Menu”“Navigation”“Skip to content”。
2. 常见交互提示，如“Back to top”“上一篇”“下一篇”。
3. Cookie/隐私/版权类通用提示。
4. 低信息量的重复分隔文本。

限制：

1. 不做基于站点定制的规则。
2. 不尝试语义级去重。
3. 不删除中长正文段落，即使其中含少量界面文本。

### 3. 递归抓取场景

递归抓取已经通过 `merge_documents()` 合并多个页面内容。本次不调整其合并逻辑。

导出行为保持简单：

1. 如果 `original_content` 是多页面合并结果，则“原文正文”直接输出该合并后的完整内容。
2. 不额外重新拆分页面小节。
3. 保留现有多页面标题与来源格式。

### 4. 配置策略

本次不新增配置开关。

原因：

1. 用户已明确希望默认输出完整网页正文。
2. 当前问题是默认行为不符合预期，不需要先引入可选配置来保守兼容。
3. 减少配置项可以保持最小实现。

如果未来出现文件体积、阅读偏好或兼容性需求，再考虑增加 `output.include_original_content` 一类配置。

## 受影响文件

1. `src/exporter/__init__.py`
   - 调整 Markdown 与 Obsidian 导出结构
   - 新增原文正文格式化逻辑
2. `tests/test_exporter.py`
   - 更新默认导出断言
   - 增加原文正文与格式化规则测试

## 测试设计

### 1. 导出结构测试

验证默认导出包含：

1. 标题
2. 来源 URL
3. 学习笔记区块
4. `## 原文正文` 区块
5. 原始正文内容

### 2. Markdown 格式化测试

至少覆盖：

1. 多余空行被规整。
2. 明显噪音行被移除。
3. 正常正文段落被保留。
4. 已有 Markdown 列表或代码块不会被破坏。

### 3. 空正文测试

当 `original_content` 为空字符串时：

1. 不输出“原文正文”章节。
2. 现有学习笔记导出仍正常。

## 错误处理

1. 若原文格式化过程中发生异常，导出器应优先保证导出成功。
2. 可降级为输出未经复杂处理的 `original_content.strip()`，而不是让整个导出失败。
3. 不影响现有 `ExportResult` 错误处理语义。

## 非目标

本次不包含：

1. 更换网页正文提取算法。
2. 引入新的 HTML to Markdown 依赖。
3. 重新设计摘要提示词。
4. 针对特定网站做定制适配。

## 实现任务

1. 在 `src/exporter/__init__.py` 中为两个 exporter 增加“原文正文”输出。
2. 在 exporter 内实现轻量正文 Markdown 格式化函数。
3. 更新 `tests/test_exporter.py`，覆盖默认导出、空正文、降噪和结构保留行为。
4. 运行相关测试验证导出行为未回归。
